FREQUENTLY ASKED QUESTIONS

OpenAI vs other LLMs — why should we build on OpenAI instead of open-source models?

OpenAI offers the most mature API ecosystem in the market: Assistants API with persistent threads, function calling that actually works, fine-tuning infrastructure, and GPT-4o's multimodal capabilities. For most enterprise use cases, the 6-12 months you'd spend engineering equivalent infrastructure on open-source models costs more than the API fees. We recommend OpenAI for speed-to-production and reliability. For data sovereignty or cost-sensitive workloads, we also deploy on Azure OpenAI and open-source alternatives — we're not dogmatic. The right model for the right job.

What about data privacy? Does OpenAI train on our data sent through the API?

OpenAI does not train on data submitted through the API — this is contractual and documented in their API data usage policies. For enterprises with stricter requirements, we deploy through Azure OpenAI Service, which provides private networking, data residency in your chosen region, and enterprise-grade compliance certifications (SOC 2, HIPAA, ISO 27001). We also implement data masking at the API gateway layer so sensitive fields never leave your environment. Your data stays yours.

How do you control OpenAI API costs at scale? We've heard horror stories about runaway bills.

Cost control is built into our architecture, not bolted on after the bill arrives. We implement: (1) token budgeting per user/session/department with hard caps, (2) model routing — GPT-4o-mini for routine tasks, GPT-4o for complex reasoning, (3) semantic caching that avoids redundant API calls, (4) prompt compression to reduce token counts without quality loss, (5) real-time cost dashboards with alerts. Average client saves 40-60% on API costs within the first month of our optimisation. No surprises.

Is fine-tuning OpenAI models actually worth it? We get decent results with prompt engineering.

Fine-tuning delivers ROI when: (1) you have domain-specific terminology the base model consistently gets wrong, (2) you need consistent output formats across thousands of requests, (3) latency matters and you want shorter prompts, (4) you want to use GPT-4o-mini for tasks that would otherwise require GPT-4o — fine-tuned mini often matches base 4o at a fraction of the cost. We've seen fine-tuned GPT-4o-mini reduce token usage by 70% versus long prompt chains on GPT-4o, with equal accuracy. We only recommend it when the data supports it — not because it sounds impressive.

What happens when the OpenAI API goes down? How do you ensure reliability?

We architect for resilience from day one. Our production OpenAI deployments include: (1) exponential backoff with jitter on all API calls, (2) circuit breakers that stop requests when failure rates spike, (3) graceful degradation — the system stays functional even when the API is slow, (4) multi-model fallback: if GPT-4o is unavailable, the system routes to GPT-4o-mini or Azure OpenAI automatically, (5) request queuing with dead-letter handling so no request is lost. Our enterprise deployments maintain 99.7%+ effective uptime across the API layer. The OpenAI status page is not your SLA — our architecture is.

Are we locking ourselves into OpenAI? What if we want to switch later?

We design for portability. Our API abstraction layer separates your business logic from the model provider — your application talks to our gateway, and the gateway routes to OpenAI, Azure OpenAI, or any other provider. If you decide to switch, you change the routing configuration, not your application code. For Custom GPTs and Assistants, we document every prompt, every function definition, every knowledge file — nothing is a black box. You own the IP. No proprietary formats. No hidden dependencies. We measure success by whether your team can operate independently within 90 days.
